What's Already Excellent

Strong Foundation

Before listing what to add, it's worth recognizing that this app is already far beyond what 99% of hospitality operators have. This is a real journey-first architecture, not a department org chart pretending to be a workflow.

✓ Journey-First Architecture

11 stages follow the emotional arc of the couple, not internal departments. Discovery → Evangelist is a complete lifecycle loop — not a linear funnel that ends at checkout.

✓ Loop-Back Design

Evangelist feeds back to Discovery (referrals) and Rebooking. Rebooking loops into Conversation and Commitment. The flywheel is architected, not bolted on.

✓ Exception Paths Are Defined

Didn't Book, Sales No-Show, Cancellation/Recovery, Arrival No-Show, and Service Recovery all have their own process branches with specific steps. Most hotels don't even name these scenarios.

✓ Actor Taxonomy Is Clean

20 actors split across Staff (13), System (6), and Guest (1). Each process step names the responsible actor. No ambiguity about who does what.

✓ Gifting & Proposal Sub-Journeys

These are sophisticated branches that most hospitality ops tools couldn't even model. The two-track buyer/recipient split in Gifting is particularly well-thought-out.

✓ System Actors Named Explicitly

PMS, CRM, Akia, Payments, Website are all treated as actors with specific responsibilities — not black boxes. This is the foundation for automation accountability.

18 Specific Upgrades

The Build List

Organized from highest-impact structural upgrades to feature-level enhancements. Each one is designed to turn bf-flow from a journey map into an operational command center.

01

Add Automation Status to Every Process Step

Tag each of the ~140 steps as: Manual (human does it), Automated (system does it), Semi-Auto (system prepares, human confirms), or Broken/Planned (should be automated but isn't yet). This is the single most important upgrade for scaling to Location #2. You can't clone what you can't classify. Display as a color-coded icon next to each step — green for automated, orange for semi, red for manual, gray for broken.

Critical
02

Add SLA / Timing Data per Step

Every process step needs a timing expectation: "Housekeeping: Prepare Villa" → SLA: Complete by 2:00 PM day of arrival. "Setter: Chase Lead" → SLA: Within 5 minutes of inquiry. This converts the journey map from a diagram into a performance contract. The BOLT Brain AI needs these numbers to trigger escalations. Display as a small timing badge below each step.

Critical
03

Add Stage Transition Triggers

The green transition banners between stages (e.g., "Commitment → Anticipation: Reservation confirmed and arrival date is in the future") need to become system-event specifications. Define the exact trigger: which system fires it, what field changes, and what webhook or automation runs. Example: Trigger: PMS status = "confirmed" AND checkin_date > today → Move HubSpot deal to Anticipation stage → Fire Akia pre-arrival sequence → Create ClickUp prep tasks. Without these, transitions are described but not enforced.

Critical
04

Add KPIs / Success Metrics per Stage

Each stage header should show its key performance indicators. Examples: Discovery: Cost per lead, attribution accuracy %. Conversation: Call-to-quote rate, quote-to-deposit rate, average days in stage. The Renewal: NPS score, rebooking conversion %, average folio total. Evangelist: Referral rate, UGC pieces generated. These make bf-flow a dashboard, not just a diagram.

High
05

Split Steps Into Guest-Facing vs. Back-Office Lanes

Right now, "Guest Couple: Communicate Preferences" sits next to "PMS: Manage Villa Inventory" in the same flat list. Add two visual swim lanes: Guest Touchpoint (what the couple sees/feels) and Backstage Operations (what happens behind the curtain). The Anticipation stage has 21 steps — a new hire can't tell which ones the guest knows about. This also maps directly to the "Moments of Truth" framework from your playbook.

High
06

Add Emotional Target per Stage

Each stage card should include the intended emotional state of the couple. Discovery: "Intrigue & aspiration." Anticipation: "Building excitement, feeling known." Arrival & Exhale: "Awe, relief, exhale." The Renewal: "Vulnerable, open, reconnected." This is what makes Bolt Farm a transformation company, not a hotel. The emotional target is the north star that every process step should serve. If a step doesn't contribute to the emotion, question whether it belongs.

High
07

Add Data Flow / Field Mapping Layer

"PMS: Sync Deal Properties to HubSpot" appears 8 times across stages. But which fields? Which direction? What happens if it fails? Add an expandable data-flow spec to every system step: Source: StayNTouch reservation.status → Target: HubSpot deal.dealstage → Mapping: "confirmed" = "Closed Won" → Failure Mode: Retry 3x, then Slack alert to #ops-alerts. This is the technical layer that makes BOLT Brain possible.

High
08

Add Live Status / Health Indicators

Connect bf-flow to your actual systems so each stage shows real-time counts: Discovery: 47 active leads this week. Anticipation: 12 arrivals in next 7 days. The Renewal: 8 couples on property. This converts the app from documentation into an operational dashboard. Even a simple daily snapshot (pulled from HubSpot + PMS APIs) would transform how leadership uses this tool.

High
09

Clean Up Ghost Actors

Concierge and Host are defined in the Actors panel but appear in zero process steps. Either document their responsibilities (and they become visible in the journey) or merge them into existing roles. HubSpot CRM is listed separately from CRM — are these different systems? If HubSpot IS the CRM, consolidate into one actor. Also: Stripe is a separate actor from Payments — same question. Every ghost actor is confusion for Location #2's team.

Medium
10

Add Process Step Dependencies (Sequential vs. Parallel)

In Anticipation, does "Housekeeping: Prepare Villa" have to wait for "Maintenance: Complete Pre-Arrival Work Orders"? Or can they run in parallel? The current flat numbered list implies sequential execution, but most of these steps are parallel workflows. Add dependency markers: → Sequential (step 12 must finish before 13) vs. ‖ Parallel (steps 9-11 can run simultaneously). This is critical for AI task dispatch optimization.

High
11

Add Ownership / RACI for Each Step

Each step names an actor (the Doer), but who's Accountable (owns the outcome), who's Consulted (input needed), and who's Informed (needs to know)? Example: "Front Desk: Conduct Rebooking Conversation" — Doer: Front Desk. Accountable: Immersion & Revenue Lead. Consulted: PMS (for availability). Informed: CRM (for deal creation). Without RACI, you have task lists without accountability chains.

High
12

Add Villa-Type Variations

The journey assumes all 34 (→50) units follow the same process. But a Treehouse arrival is different from a Dome arrival is different from a Mirror House arrival. Add a "Villa Type" filter or variation layer so you can see: Dome-specific prep steps in Anticipation, Treehouse-specific safety briefing at Arrival, Mirror House-specific photo opportunity during Renewal. This captures the operational reality that each structure type has unique workflows.

Medium
13

Add "Dream Weaver Budget Tracker" to Anticipation & Renewal

The $250 discretionary Dream Weaver spend per guest is one of your most powerful differentiators, but it doesn't appear anywhere in bf-flow. Add a process step under Anticipation: "Dreamweaver: Allocate Personalization Budget" and under Renewal: "Dreamweaver: Execute Surprise & Delight Moments." Track the budget allocation, spend, and what was delivered — this data becomes the training set for BOLT Brain's personalization engine at scale.

Medium
14

Add Reconnection Profile Capture to Anticipation

Your Institutional AI report designed a "Reconnection Profile" — the deep personalization quiz that captures relationship stage, connection style, and transformation readiness. This should be a named process step in Anticipation: "Akia/CRM: Deliver Reconnection Profile Quiz" → "BOLT Brain: Generate Personalization Playbook." Currently, "Guest Couple: Communicate Preferences" is too vague to capture this transformational data point.

Medium
15

Add Revenue Attribution per Stage

Each stage generates or influences revenue differently. Conversation: Deposit conversion. Anticipation: Upsell revenue (spa, meals, add-ons). The Renewal: On-site rebooking deposits, folio charges. Afterglow: Referral-attributed bookings. Tag each step that directly generates revenue and add a running $ estimate per stage. This makes bf-flow a revenue attribution model, not just a process map.

Medium
16

Add a "Reconnection Circle" Post-Stay Stage

Your Institutional AI report proposed a $49/month membership for post-stay relationship wellness. The current Afterglow stage covers testimonials and marketing sequences, but there's no stage for ongoing subscriber engagement. Add a stage between Afterglow and Rebooking (or nest it within Afterglow) for: monthly content delivery, community engagement, exclusive rebook offers, anniversary milestone triggers. This is the $300K-$600K ARR opportunity that lives outside the current journey.

Future
17

Add Version History / Changelog

As this journey evolves (and it will — every week), you need to know what changed, when, and why. Add a simple version log: "v3.2 — Added Spa Team to Arrival stage, removed duplicate Didn't Book branch from Inquiry." This is essential for training new staff ("Why did we change this?"), debugging process failures ("When did this step get removed?"), and Location #2 onboarding ("What's the canonical v4.0 journey?").

Future
18

Add Multi-Location Support

When Location #2 opens, will it use the same 11-stage journey? Almost certainly — but with location-specific variations in villa types, staff roles, local experiences, and vendor integrations. Add a location selector to bf-flow now (even if there's only one location) so the architecture supports: Shared core journey (same for all locations) + Location overrides (different spa treatments, different meal vendors, different villa prep procedures). Build the multi-tenant architecture before you need it.
