1.3 Sync Architecture — Current vs. Target

CURRENT STATE: Inline Fire-and-Forget

Server Action → DB Write → syncBookingToHubspot() (inline, async, non-blocking) → Success logs "synced" / Failure logs "hubspot_sync_failed" with NO automatic retry. Recovery is manual only via AI tool.

TARGET STATE: Event-Driven with Upstash QStash

Server Action → DB Write → publish("booking.status_changed") → QStash queues → POST /api/events/ → Verify HMAC signature → syncBookingToHubspot() → On failure: 3x exponential backoff → Dead Letter Queue → Alert. ~35 events/day, well within free tier (500/day).

6 OPEN DECISIONS BLOCKING MIGRATION

1. QStash unavailability fallback (inline fallback vs. log-only). 2. Retry delay intervals unspecified. 3. DLQ alert channel undefined (Slack? Email? Dashboard?). 4. Consumer endpoint auth mechanism not detailed. 5. triggeredBy payload shape not ratified. 6. Phase 1 Dual-Write status unknown — is publish() running today?

Quick Wins — Implementable in ≤ 2 Weeks

1. Sync last_stay_date on Checkout

Add one property write to syncBookingToHubspot(): write booking.checkOut to last_stay_date when status = checked_out. Unblocks the entire Past Guest pipeline's time-based automations (14-day, 90-day, COLD stages).

Owner: PMS Dev Effort: 2 hours Impact: Critical — fixes a broken pipeline

2. Fix Rebooking Stage ID Map

PMS currently maps confirmed → 1308814489 (Reservation Closed Won) globally. For rebooking deals, the target must be 998541953 (Rebooking Closed Won). Add pipeline context to HUBSPOT_STAGE_MAP lookup.

Owner: PMS Dev Effort: 4 hours Impact: High — deals landing in wrong stage

3. Add deal_source Enum Value

Add "Online Booking" to the HubSpot deal_source enumeration. Guest self-serve bookings currently can't be attributed correctly because "online_booking" isn't a valid value.

Owner: HubSpot Admin Effort: 15 minutes Impact: High — fixes attribution for all self-serve bookings

4. Auto-Delete Expired Upsell/Rebooking Deals

Replace the manual admin task (Kara deletes expired deals) with a HubSpot workflow: if deal is in Upsell or Rebooking pipeline AND check-out date < today AND stage = Expired → auto-archive after 7 days. Save ~1 hour/week of admin time.

Owner: HubSpot Admin Effort: 1 hour Impact: Medium — eliminates recurring busywork

5. Retrieve All Missing Stage IDs

Run GET /crm/v3/pipelines/deals/{pipelineId}/stages for Upsell (873728437), Recover (873783453), and Rebooking (873057543). Fill in the 13+ missing stage IDs in the spec. This is blocking all future PMS integration work on these pipelines.

Owner: PMS Dev / HubSpot Admin Effort: 30 minutes Impact: High — prerequisite for Phase 2

6. Smart Review Solicitation via Akia

Replace the manual "staff asks at checkout" with: Departure NPS pulse (Akia SMS at checkout) → score ≥ 9 = auto-send review link (rotate TripAdvisor/Google/Airbnb). Score < 8 = route to service recovery. Never ask an unhappy guest for a public review.

Owner: Guest Experience Lead Effort: 3-4 hours (Akia workflow) Impact: High — protects 4.9★ average + increases review volume

7. Pre-Arrival Spa Upsell SMS

Add a Day -7 Akia SMS with spa menu + booking link. Currently the spa upsell only fires at 11am during the stay. Pre-booked spa has higher show rate, enables staffing, and captures guests while they're still in "planning mode."

Owner: Guest Experience + Spa Effort: 2 hours (Akia template) Impact: High — move spa booking upstream = more revenue per stay

8. Guest Request → ClickUp Auto-Task

Configure Akia → ClickUp integration: every guest text request auto-creates a ClickUp task with guest name, unit number, request text, and 15-min SLA timer. Auto-assign to on-duty team member. Auto-escalate to manager at 10 min.

Owner: Operations Lead Effort: 4-6 hours Impact: Critical — prevents the "dropped foil" scenario

9. Rename stayntouch_reconciled → pms_reconciled

Simple HubSpot property rename + workflow update. Eliminates confusion from the dead StayNTouch brand name appearing in daily staff workflows. Takes 30 minutes but reduces cognitive noise across every reconciliation touchpoint.

Owner: HubSpot Admin Effort: 30 minutes Impact: Low but symbolic — clean house

10. DLQ Alert Channel Decision

Make the open decision: Dead Letter Queue alerts go to #ops-alerts in Slack. This unblocks the QStash migration. Also decide: retry intervals should be 10s / 60s / 300s (aggressive recovery for a low-volume system with ~35 events/day).

Owner: PMS Dev + Ops Lead Effort: 1 meeting Impact: Medium — unblocks sync architecture migration
SECTION 6

Scalability Red Flags — What Breaks at 50+ Units or Property #2

# Red Flag Why It Breaks Fix Required Priority
1 Single-Deal-Per-Booking Architecture The PMS currently stores one hubspotDealId per booking. But HubSpot creates 3-5 deals per booking lifecycle (Reservation + Upsell + Past Guest + Rebooking + potentially Recover). At 50 units, you'll have ~250 deals/month that the PMS can't track Build the booking_deals join table (Phase 2). This is the single most important architectural change. Without it, the PMS is blind to 60%+ of HubSpot deals CRITICAL
2 Fire-and-Forget Sync with No Retries At 34 units, ~35 sync events/day with occasional failures is manageable via manual recovery. At 50 units + second property, you're looking at 80-100+ events/day across two PMS instances. Manual recovery becomes a full-time job Complete the QStash migration. The architecture is designed and documented. The 6 open decisions need to be made and Phase 1 (Dual-Write) needs to be verified or implemented CRITICAL
3 Recover Pipeline is 90% Manual 5 of 6 stage IDs are missing. 7 sub-procedures are tribal knowledge. Sales team drives most stages manually. At scale, cancellation volume increases proportionally — you can't scale manual recovery calls Phase 2: PMS recovery wizard. Staff selects one of 7 outcomes → PMS auto-fills correct properties → auto-moves to correct stage. Also: retrieve all 6 stage IDs and document the full flow HIGH
4 Dream Weaver System is Person-Dependent The Dream Weaver team uses a $250 discretionary budget per guest to orchestrate surprises. This works with a small, dedicated team that knows every guest. At 100+ guests/day across 2 properties, the "know every guest" model collapses Systematize the Dream Weaver: AI Personalization Engine (quiz) → GPT classification → auto-generated ClickUp checklist per guest. Human execution of surprise moments, but AI-planned logistics. The budget, categories, and timing rules should be in the system, not in someone's head HIGH
5 Arrival Protocol Requires 3+ Staff per Check-In Welcome Wows: greeted by name at car, lavender towel, luggage service, QR check-in, champagne, chair massage, guided room tour. This is 60-90 minutes of coordinated staff effort per arrival. At 15+ check-ins/day (50 units), you need a dedicated arrival team of 5-6 people Tiered arrival protocol: Signature Welcome (full protocol) for first-time guests. Express Welcome (QR check-in + welcome kit in room + self-guided tour with audio) for returning guests. Use PMS guest history to auto-assign tier MEDIUM
6 No Multi-Property HubSpot Architecture All 7 pipelines are "Mountain-" prefixed. Pipeline IDs are hardcoded. Property names assume a single property. There's no namespace for a second location Design the multi-property schema now: either separate pipeline sets per property (Mountain-Reservation, [NewSite]-Reservation) or add a property_location custom property to every deal. The HubSpot architecture decision must happen before property #2 breaks ground HIGH
7 Nurturing Pipeline Creates Infinite Deals Every CL-Nurturing stage creates a NEW deal. If a lead is lost 5 times, you have 5 nurturing deals. At scale with thousands of leads, the pipeline becomes a deal graveyard with no staleness check or cap Add a max-iteration rule: if a contact has ≥ 3 CL-Nurturing deals, auto-move to CL-Not Qualified instead of creating another deal. Also add a staleness check: if a nurturing deal has been in Follow-Up Needed for > 90 days with no activity, auto-archive MEDIUM
8 QStash Free Tier Ceiling Free tier = 500 messages/day (retries count). At 50 units you're at ~80 events/day. At 2 properties with Phase 3 consumers (email, housekeeping, Slack, analytics), you could hit 300-400 events × 3 retries = 1,200 messages/day. Over the free tier Budget for QStash Pro ($1/day for 10K messages). This is a rounding error in your P&L but must be planned before you hit the wall. Also: optimize retry strategy to minimize retry volume LOW

THE BOTTOM LINE

Bolt Farm's guest experience is world-class. The reviews prove it. The product-market fit is undeniable — couples come to disconnect, reconnect, and leave transformed. What's holding you back isn't the product. It's the plumbing.

Three moves unlock everything: (1) Sync last_stay_date — one line of code that unblocks the entire post-stay revenue engine. (2) Build booking_deals — the join table that lets your PMS see the full deal universe. (3) Migrate to QStash — the event bus that ensures no sync ever silently fails again.

Everything else — the Dream Weaver systemization, the revenue upsell automation, the multi-property architecture — builds on those three foundations. Fix the plumbing. The magic is already there.
